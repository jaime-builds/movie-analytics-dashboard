# Movie Analytics Dashboard Audit - 2026-07-05

## Executive Summary

This was a static, repo-only audit of the Flask app, database models and migrations, templates, static assets, tests, CI, deployment files, and committed docs. I did not implement fixes, refactor code, update tests, or run runtime verification.

Overall, the app is feature-rich and has meaningful test coverage, but several issues should be addressed before new feature work. The biggest risks are security/auth correctness, database/migration drift, and production resilience. In particular, an authenticated page is cached as shared rendered HTML, state-changing routes lack CSRF protection, user-controlled collection names can be inserted as HTML, Alembic and `create_all` initialization paths disagree, and some public API/query paths can fail or become expensive under bad inputs.

Recommended next step: fix the security and migration-health items first, then add focused regression coverage around auth, caching, API validation, and migrations before continuing feature development.

## Highest Priority Findings

- Shared caching on `/analytics` can expose login-specific rendered HTML across users.
- State-changing authenticated routes have no CSRF protection, and `/logout` is a GET request.
- Collection names are inserted into `innerHTML` on the movie detail page, creating a stored/self XSS path.
- Database initialization is split between SQLAlchemy `create_all` and Alembic migrations, with at least one migration that likely needs SQLite batch handling.
- API pagination and rate-limit exemptions leave public endpoints vulnerable to errors and expensive requests.
- TMDB calls have no HTTP timeouts and are made synchronously from user-facing detail pages.

## Findings

### [High] Shared cached analytics page can leak user-specific rendered HTML

- Area: Security, auth, caching, UX
- Files involved: `src/app.py:1382-1499`, `templates/base.html:94-133`, `config/config.py:54-66`
- What I found: The `/analytics` route is decorated with `@cache.cached(timeout=3600)` and renders templates with `current_user=user`. The shared base template conditionally renders authenticated navigation, username, and user-menu links based on `current_user`. The cache key is not varied by user or auth state.
- Why it matters: The first rendered `/analytics` response for a logged-in user can be reused for other users or anonymous visitors for up to an hour. At minimum, this can display the wrong username/navigation state; depending on future template changes, it could expose more user-specific content.
- Recommended fix: Do not cache the full authenticated HTML response. Cache only expensive aggregate data, vary the cache key by user/auth state, or skip caching when `session["user_id"]` is present.
- Suggested tests: Add a regression test that requests `/analytics` as one logged-in user, then as a different user or anonymous client, and asserts the navbar/user-specific content does not leak.
- Implementation risk: Medium. The safest fix is to move caching down to aggregate query helpers, which requires care to preserve analytics performance.

### [High] Authenticated state-changing routes lack CSRF protection

- Area: Security and auth
- Files involved: `src/app.py:374-650`, `src/app.py:2738-2896`, `templates/login.html:14`, `templates/register.html:14`, `templates/movie_detail.html:320`, `templates/collections.html:56-89`, `templates/collection_detail.html:29`, `requirements.txt`, `tests/conftest.py:33`
- What I found: Favorites, watchlist, ratings, reviews, and collection create/delete/add/remove routes accept POST requests without CSRF tokens. `/logout` is a GET route. The repo does not include a CSRF extension in `requirements.txt`, and test config disables `WTF_CSRF_ENABLED` even though no CSRF layer appears to be active.
- Why it matters: Any site can cause a logged-in browser to submit state-changing requests. Combined with the collection-name HTML injection issue below, this increases account-level risk.
- Recommended fix: Add CSRF protection for forms and JavaScript requests, require POST for logout, include CSRF headers in fetch helpers, and add server-side validation for all state-changing endpoints.
- Suggested tests: Add tests that POST without a CSRF token and expect rejection, tests that valid forms/fetches include tokens, and a test that GET `/logout` no longer changes session state.
- Implementation risk: Medium. Templates and JavaScript fetch calls need coordinated updates, but the route surface is finite.

### [High] Collection names can be inserted into HTML on the movie detail page

- Area: Security, frontend
- Files involved: `src/app.py:2747-2762`, `src/app.py:2916-2927`, `templates/movie_detail.html:770-776`, `templates/movie_detail.html:787-790`
- What I found: Collection names are user-controlled and are returned directly from `/api/v1/collections`. The movie detail page builds collection dropdown items with `innerHTML` and interpolates `c.name` into the HTML string.
- Why it matters: A malicious collection name can execute script when the collection dropdown is rendered. This is especially serious because the app currently lacks CSRF protection, so an attacker may be able to force creation of a hostile collection name for a logged-in user.
- Recommended fix: Build dropdown items with DOM APIs and `textContent`, or escape collection names before insertion. Also validate or normalize collection names server-side as a defense-in-depth measure.
- Suggested tests: Add a browser-level or template/JS regression test using a collection name like `<img src=x onerror=alert(1)>` and assert it is rendered as text, not executable HTML.
- Implementation risk: Low. The main fix is localized to the dropdown rendering code, though CSRF should also be addressed.

### [High] Database initialization path is split between `create_all` and Alembic

- Area: Database and migrations, deployment, documentation
- Files involved: `src/models.py:327-330`, `README.md:29`, `README.md:365`, `Dockerfile:53`, `migrations/versions/001_initial_schema.py`, `migrations/versions/002_expand_password_hash.py:24-31`
- What I found: Local setup documentation tells developers to initialize the database with `python -m src.models`, which calls `Base.metadata.create_all(engine)`. Docker startup runs `alembic upgrade head`. The second migration alters `users.password_hash` from `String(128)` to `String(256)` with `op.alter_column`, which likely needs SQLite batch handling. This needs runtime verification on a clean SQLite database.
- Why it matters: A database created through `create_all` is not stamped with Alembic revision state. Running Alembic later may try to recreate existing tables or fail on SQLite column alteration. This can break local onboarding, test database setup, and deployment recovery.
- Recommended fix: Make Alembic the single schema initialization path. Update local setup docs to run `alembic upgrade head`, remove or clearly demote `create_all` initialization, and convert SQLite-sensitive migrations to `batch_alter_table` where needed.
- Suggested tests: Add a CI job or test script that creates a clean SQLite database with Alembic from base to head, and another that upgrades from migration `001` to head.
- Implementation risk: Medium. Existing local databases may need clear migration/stamping guidance.

### [Medium] Login `next` handling is both incomplete and open to external redirects

- Area: Security and auth, UX
- Files involved: `src/app.py:349-360`, `templates/login.html:14`, `tests/test_auth.py:204-214`
- What I found: After login, the app redirects directly to `request.args.get("next")` without validating that it is local. However, the login form posts to `url_for('login')`, which drops the original `?next=...` query from the common GET-login-form flow. Tests cover a direct POST to `/login?next=/analytics`, but not the normal GET `/login?next=...` form submission or external URLs.
- Why it matters: The intended redirect-after-login behavior can fail for real users, and direct POSTs with an external `next` value can become an open redirect.
- Recommended fix: Preserve `next` through the form with a hidden field or action URL, and validate that redirects are relative/local before using them.
- Suggested tests: Add tests for GET `/login?next=/favorites` followed by form POST, and for malicious values such as `https://example.com`.
- Implementation risk: Low.

### [Medium] API pagination accepts invalid low values and can raise errors

- Area: Bugs and correctness, API resilience
- Files involved: `src/app.py:1686-1758`, `src/app.py:1856-1900`, `src/app.py:2011-2013`, `src/app.py:2114-2148`, `src/app.py:936-1000`, `src/app.py:3305-3381`
- What I found: Several endpoints cap maximum page size but do not enforce a minimum. For example, `/api/v1/movies?per_page=0` can reach total-page calculation with division by zero. Negative `per_page`, negative `page`, and negative `limit` values can also produce confusing offsets or empty/expensive queries. Similar max-only validation appears in movie search, actors, top movies, and some HTML list/search routes.
- Why it matters: Public endpoints can return 500s or perform unexpected queries for simple malformed input.
- Recommended fix: Centralize pagination parsing so `page >= 1`, `1 <= per_page <= max`, and `1 <= limit <= max`. Return a clear 400 for invalid values or clamp consistently.
- Suggested tests: Add API tests for `per_page=0`, `per_page=-1`, `page=0`, `page=-5`, and `limit=-10` across the main list endpoints.
- Implementation risk: Low to Medium. Behavior for invalid inputs must be chosen consistently.

### [Medium] Main public movie API is exempt from rate limiting

- Area: Performance and efficiency, security/abuse protection
- Files involved: `src/app.py:1679-1758`, `src/app.py:1728`, `src/app.py:1749`, `README.md:71`
- What I found: `/api/v1/movies` has `@limiter.exempt`. The route performs `query.count()`, supports multiple filters/sorts, and serializes related genre data for each returned movie. README documentation says API and analytics endpoints are rate-limited.
- Why it matters: This endpoint is public and can be used for repeated expensive queries. It also contradicts the repo's own documented security/performance posture.
- Recommended fix: Remove the exemption or apply a route-specific limit. Keep page-size caps and add invalid-input guards before doing database work.
- Suggested tests: Add a focused rate-limit test for `/api/v1/movies`, with rate limiting enabled in the test config.
- Implementation risk: Low.

### [Medium] TMDB calls have no HTTP timeouts and run synchronously on detail pages

- Area: Performance and operations
- Files involved: `src/tmdb_api.py:24`, `src/app.py:137-140`, `src/app.py:1310-1315`, `scripts/sync_tmdb_data.py:75`
- What I found: The TMDB client uses `requests.get(url, params=params)` without a timeout. Movie detail pages synchronously call TMDB for trailers and watch providers when a `tmdb_id` exists. The sync script also performs requests without explicit timeouts.
- Why it matters: A slow or hanging TMDB response can tie up web workers and make movie detail pages unreliable. For production, this creates avoidable availability risk.
- Recommended fix: Add connect/read timeouts, log failures consistently, and cache or persist trailer/provider metadata instead of calling TMDB on every detail-page request.
- Suggested tests: Mock `requests.get` to raise timeout exceptions and assert the app returns the movie detail page with graceful fallback content.
- Implementation risk: Medium. Timeouts are easy; moving detail-page TMDB data into cached/background sync needs more design.

### [Medium] Several high-traffic query paths are missing supporting indexes

- Area: Database and performance
- Files involved: `src/models.py`, `migrations/versions/001_initial_schema.py`, `database/schema.sql:93-98`, `src/app.py:720-733`, `src/app.py:1154-1158`, `src/app.py:1223-1225`, `src/app.py:1293`, `src/app.py:2073-2086`, `src/app.py:2121-2125`, `src/app.py:2554-2583`, `src/app.py:3065-3073`, `src/app.py:3124-3215`
- What I found: The committed model/migration indexes do not cover several heavily used joins and sorts, including `cast.movie_id`, `cast.person_id`, `crew.movie_id`, `movie_companies.company_id`, and common movie sort/filter fields such as `release_date`, `vote_average`, and `popularity`. The legacy `database/schema.sql` documents some indexes that are absent from the active SQLAlchemy model and Alembic migration path.
- Why it matters: Actor pages, common-film discovery, company/director pages, analytics, and public APIs can become slow as data grows.
- Recommended fix: Add Alembic migrations for the missing indexes after confirming query plans against representative data. Keep model `Index(...)` declarations and migrations aligned.
- Suggested tests: Add query-plan or performance smoke checks for representative actor, company, and movie-list queries, or at least migration tests that assert expected indexes exist.
- Implementation risk: Medium. Index names and existing production database state need to be checked before migration.

### [Medium] Review uniqueness is enforced only in application code

- Area: Database correctness
- Files involved: `src/models.py:195-213`, `src/app.py:604-620`
- What I found: The `Review` model has indexes on `movie_id` and `user_id`, but no unique constraint for `(user_id, movie_id)`. The route checks for an existing review and updates it, but the database does not prevent duplicates during races, manual imports, or future code paths.
- Why it matters: Duplicate reviews can distort displayed review counts and user profile data, and cleanup becomes harder after duplicates exist.
- Recommended fix: Add a unique constraint on `(user_id, movie_id)` after checking/merging existing duplicates. Consider matching the rating table's uniqueness pattern.
- Suggested tests: Add a database-level test that two reviews for the same user/movie cannot be inserted.
- Implementation risk: Medium if production data may already contain duplicates; Low otherwise.
